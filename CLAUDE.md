## Commands

```bash
# Development (hot reload at localhost:3000)
cargo leptos watch

# Release build (outputs to target/server/release/ and target/site/)
cargo leptos build --release

# E2E tests (run npm install in end2end/ first)
cargo leptos end-to-end

# Format (use leptosfmt for component macros, rustfmt for the rest)
leptosfmt src/
cargo fmt
```

**Prerequisites:**
- `cargo install cargo-leptos --locked`
- `rustup target add wasm32-unknown-unknown`
- `npm install -g sass`
- `npm install` inside `end2end/`
- `.env` file with `DATABASE_URL=<postgres_connection_string>`

**Database migrations:**
```bash
diesel migration run
diesel migration revert
```

## Architecture

**Full-stack Rust** using Leptos 0.8 (reactive UI, similar to SolidJS) + Axum 0.8 (HTTP server). The same codebase compiles to both server binary (SSR feature) and WASM bundle (hydrate feature). `cargo-leptos` orchestrates both compilation targets.

### Feature flags
- `ssr` ‚Äî server-only code: Axum, Diesel, database access
- `hydrate` ‚Äî client-only code: WASM hydration, console_error_panic_hook
- Guard server-only code with `#[cfg(feature = "ssr")]`

### Routing (`src/app.rs`)
- `/` ‚Üí `HomePage` ‚Äî lists all queues
- `/queue/:url_name` ‚Üí `QueuePage` ‚Äî queue details with rows
- `/add` ‚Üí `AddQueuePage` ‚Äî create new queue

### Server functions (`#[server]` macro)
All client‚Üîserver communication goes through Leptos server functions ‚Äî they look like normal async Rust functions but auto-serialize as HTTP RPC. Client calls them directly; the framework handles serialization. All are defined co-located with the components that use them.

### Data flow
1. `#[server]` functions call `db::api` functions
2. `db::api` functions operate on `db::models` (Diesel structs) and convert to domain types from `queue.rs`
3. Domain types (`QueueInfo`, `QueueEntry`, `EntryPlayers`) are shared client/server via `serde`

### State management
- `Resource::new()` / `OnceResource::new()` for async data fetching (refetches on signal change)
- `RwSignal` / `ReadSignal` / `WriteSignal` for reactive state
- `provide_context()` / `use_context::<T>()` for dependency injection down the component tree
- Key contexts: `DbPool` (server only), `QueueInfo`, `AddModalState`, `EntryStore`

### Optimistic updates (`src/pages/queue/rows/`)
New queue entries are added locally with a `LocalUuidState::Pending(temp_uuid)` before the server confirms. The `EntryStore` (Vec<`LocalQueueEntry`>) holds these pending entries. On server response, the temp UUID is replaced with the real one (`LocalUuidState::Resolved`). Rows with unresolved UUIDs are non-interactive.

### Database (`src/db/`)
- PostgreSQL via Diesel ORM (compile-time checked queries) + diesel-async for async ops
- deadpool connection pool (max 4 connections), pool stored in Axum state, extracted via Leptos `extract()`
- Schema: `queues` (id, url_name unique, display_name) ‚Üí `queue_rows` (id, queue_id FK, left_player_name, right_player_name, queue_order, UNIQUE(queue_id, queue_order), CHECK not both null)
- `src/db/schema.rs` is auto-generated by Diesel ‚Äî do not edit manually

### Styling
SCSS in `style/` compiled by dart-sass. Component-specific styles in `style/components/`, page styles in `style/pages/`, all imported into `style/main.scss`.
